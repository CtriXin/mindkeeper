# 网络方案对比（已落地版）

## 你的现状

- **OAuth**：global Claude CLI（`~/.zshrc` 中的 `claude()` 函数覆盖）
- **Proxy**：`socks5h://127.0.0.1:41002` → SSH → 服务器 → Webshare（9.142.29.190）
- **链路**：Claude CLI → 41002(SOCKS5) → SSH 加密 → 服务器 socks5_relay.py → Webshare → Anthropic OAuth

出口：9.142.29.190（Comcast Cable，住宅 IP），固定不漂移。

---

## 你问的 4 个问题

### 1. DNS 泄漏：HTTP proxy vs socks5h

| 场景 | HTTP proxy（旧 31001） | socks5h（当前 41002） |
|------|----------------------|----------------------|
| HTTPS 请求（Claude OAuth/API） | ✅ DNS 在代理端解析（CONNECT 隧道） | ✅ DNS 在代理端解析 |
| HTTP 请求（telemetry/更新） | ❌ DNS 在本地解析 | ✅ DNS 在代理端解析 |

**结论**：当前用 socks5h，DNS 全部在代理端解析，无泄漏。

### 2. 漂移问题

41002 是 SSH 本地端口转发（`-L 41002:127.0.0.1:41002`），**不会漂移**。

出口永远是 Webshare 的 9.142.29.190（除非 Webshare 更换 IP）。

Clash 的 mixed-port 7897 **会漂移**（随你切换节点而变），但 Claude 不走了。

### 3. 故障行为：不通时会怎样？

**会断联，不会 fallback 到本地流量。**

- proxy 连不上 → ECONNREFUSED → 报错退出
- 不会偷偷直连

这是 GPT 要的"不通就失败"。

### 4. 三种方案哪个最好？

| 方案 | 安全性 | 稳定性 | 复杂度 | DNS | 推荐度 |
|------|--------|--------|--------|-----|--------|
| **当前：SSH + 服务器 socks5_relay + Webshare** | ✅ 加密 | ✅ 固定住宅 IP | ✅ 简单 | ✅ socks5h | ⭐⭐⭐⭐⭐ |
| Clash + 固定规则 | ⚠️ 本地软件 | ⚠️ 会漂移 | ❌ 复杂 | ✅ 规则可控 | ⭐⭐⭐ |
| 直连服务器端口 | ❌ 暴露公网 | ⚠️ 看网络 | ✅ 简单 | ✅ 无代理层 | ⭐⭐ |

**当前方案最好**，理由：
1. SSH 加密传输
2. 不暴露服务器端口到公网
3. Webshare 住宅 IP（比数据中心 IP 更像真实用户）
4. socks5h，DNS 代理端解析
5. 故障时断开，不会 fallback

---

## 一句话结论

**保持当前方案：SSH → 服务器 socks5_relay.py → Webshare。**

不要换 Clash，不要暴露服务器端口。
