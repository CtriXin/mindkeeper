**旧方案**：MMC + proxy

为什么变了：
- MMC 是 OAuth 账户管理，但你的核心诉求是"出口稳定 + 住宅 IP"

```
  → Webshare HTTP proxy (9.142.29.190:6348)
  → Anthropic API / 中转服务
```

出口 IP：9.142.29.190（Comcast Cable，住宅 IP）

## 二、遥测：开启会暴露什么？Proxy 兜得住吗？

### 当前状态

`~/.zshrc` 中**没有**设置 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`，所以遥测是**开启的**。

### 遥测会发送什么？

| 服务 | 发送内容 | 是否走 Proxy |
|------|---------|-------------|
| **Statsig** | 使用模式（活跃时间、功能偏好、会话时长） | ✅ 走 Proxy |
| **Sentry** | 错误报告（错误栈、CLI 版本） | ✅ 走 Proxy |
| **自动更新** | 版本检查请求 | ✅ 走 Proxy |
| **`/bug` 命令** | 完整对话历史 | ✅ 走 Proxy |

**注意**：Statsig 数据不包含你的代码或文件路径，但包含"使用模式"——这可以构成设备画像的一部分（比如你习惯什么时间使用、偏好什么功能）。

### Proxy 兜得住吗？

| 流量类型 | 是否走 Proxy | 说明 |
|---------|-------------|------|
| Claude API 请求 | ✅ | `ALL_PROXY` 覆盖 |
| Statsig/Sentry | ✅ | 同属于 Claude 进程的 HTTP 请求 |
| DNS 解析 | ✅ | `socks5h`，代理端解析 |
| **MCP Servers** | ⚠️ **可能不走** | 子进程可能不继承 `ALL_PROXY` |

**MCP Servers 是潜在的泄漏点**：
- Claude 启动的 MCP server 是独立子进程
- 它们可能**不继承** `HTTP_PROXY`/`ALL_PROXY` 环境变量
- 如果 MCP server 向外发请求，可能走直连（暴露真实 IP）

### 结论

**Proxy 兜住了大部分，但不是 100%。**

如果你追求"Claude 完全不认识我"：
- ✅ Proxy 隐藏了 IP（住宅 IP）
- ✅ socks5h 隐藏了 DNS
- ⚠️ 遥测数据仍然暴露了"使用模式"（设备画像的一部分）
- ⚠️ MCP servers 可能绕过 proxy

## 三、要不要禁用遥测？

### 选项 A：禁用（推荐）

在 `~/.zshrc` 的 `claude()` 函数中添加：
```bash
CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC="1" \
```

**优点**：
- Statsig/Sentry/自动更新 全部关闭
- 减少设备画像数据
- 更符合"完全隔离"的诉求

**代价**：
- GitHub issue #45381：禁用遥测会禁用 1 小时 prompt cache，降级到 5 分钟
- 长对话中 cache 更频繁失效，可能增加 token 消耗
- 但你的中转服务（newapi-test.cmpc.fun / api.kimi.com）的 cache 机制与官方不同，影响不确定

### 选项 B：保持开启

**优点**：
- 保留 1 小时 cache
- 自动更新可用
- 错误报告帮助 Anthropic 改进 CLI

**风险**：
- 使用模式被记录（虽然 IP 是住宅 IP，但活跃时间、功能偏好仍然是一个指纹维度）

### 建议

**禁用遥测。**

理由：
1. 你的核心诉求是"完全隔离"
2. Webshare 住宅 IP 解决了 IP 层面的暴露，但使用模式仍然是指纹
3. cache 降级的影响在你的中转服务上不确定
4. 如果成本增加，可以通过更积极的 `/compact` 来缓解

## 四、MCP Servers 的泄漏风险

如果你使用了 MCP servers（如 Figma、hive、mindkeeper），它们是独立子进程。

**当前风险**：
- MCP server 可能不继承 `ALL_PROXY`
- 如果 MCP server 向外发请求（如 Figma API），可能暴露真实 IP

**缓解方案**：
1. 在 `~/.claude/settings.json` 中为 MCP server 单独设置 proxy（如果支持）
2. 或者在 `~/.zshrc` 中全局设置 proxy（已经做了）
3. 检查每个 MCP server 的配置，看是否有 proxy 选项

## 五、下一步建议

### 立即做（1 分钟）

在 `~/.zshrc` 的 `claude()` 函数中添加遥测禁用：

```bash
claude() {
    # ... 显示信息 ...
    
    HTTP_PROXY="${CLAUDE_PROXY}" \
    HTTPS_PROXY="${CLAUDE_PROXY}" \
    ALL_PROXY="${CLAUDE_PROXY}" \
    NO_PROXY="127.0.0.1,localhost,::1" \
    TZ="America/Los_Angeles" \
    LANG="en_US.UTF-8" \
    LC_ALL="en_US.UTF-8" \
    LC_CTYPE="en_US.UTF-8" \
    CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC="1" \
    command claude "$@"
}
```

### 检查 MCP Servers（5 分钟）

检查你使用的 MCP server 是否走了 proxy：
```bash
# 启动 Claude 后，在另一个终端查看进程环境
ps e | grep -i mcp | grep -oE "HTTP_PROXY=[^ ]+"
# 如果为空，说明 MCP server 没继承 proxy
```

### 观察 cache 影响（长期）

禁用遥测后，观察：
- 长对话中是否频繁重新读取文件
- 账单是否有明显增加
- 如有问题，可以再开启遥测
